# MADMEDSALES Admin 대시보드 — 완전판 기획서

> 버전: 2.0 | 작성일: 2026-02-22
> 이 문서의 목적: 화면 껍데기가 아니라, "왜 이 화면이 필요하고, 무슨 판단을 내리고, 어떤 행동으로 이어지는가"를 설계한다.

---

## 0. 이 시스템은 무엇인가

MADMEDSALES는 **1인 CEO가 AI로 수백 개 병원에 동시 영업하는 시스템**이다.

핵심 전제:
- 영업사원이 없다. AI가 영업사원 역할을 한다.
- CEO(MMAKID)는 시스템의 **컨트롤 타워**에서 흐름을 보고 판단만 내린다.
- Admin 대시보드는 그 컨트롤 타워다.

따라서 Admin의 모든 화면은 이 질문에 답해야 한다:
> **"지금 이 화면에서 내가 뭘 판단하고, 뭘 해야 하지?"**

답이 없는 화면은 만들지 않는다.

## 1. 영업 파이프라인 — 병원이 걸어가는 길

병원 하나가 "모르는 병원"에서 "계약한 고객"이 되기까지 걸어가는 길이다.
Admin은 이 길 위에 병원들이 어디쯤 있는지 보고, 막힌 곳을 뚫어준다.

```
수집(POOL)  →  분석(PROFILED)  →  리드(LEAD)  →  접촉(CONTACTED)
                                                        ↓
계약(WON)  ←  제안(PROPOSAL)  ←  데모(DEMO)  ←  반응(RESPONDED)
     ↓                                              ↓
  사후관리                                        무반응 → 장기육성
```

### 스테이지 정의

| 스테이지 | 코드 | 의미 | 진입 조건 | 머무는 기간 |
|---------|------|------|----------|-----------|
| 수집 | `POOL` | 심평원에서 가져온 전체 병원 | 시스템 초기 로딩 | 영구 |
| 분석 완료 | `PROFILED` | 웹사이트 크롤 + AI 분석 완료 | 크롤 성공 + 4축 점수 산출 | 영구 |
| 리드 | `LEAD` | 영업 대상으로 선정됨 | 종합 60점 이상 + 시그널 감지 | ~3일 |
| 접촉 | `CONTACTED` | 첫 이메일 발송됨 | 이메일 발송 성공 | ~7일 |
| 반응 | `RESPONDED` | 오픈/클릭/회신 중 하나 발생 | 추적 이벤트 감지 | ~14일 |
| 데모 | `DEMO` | 데모 신청 or 스케줄 확정 | 데모 신청 폼 제출 | ~7일 |
| 제안 | `PROPOSAL` | 견적서/제안서 발송됨 | 관리자가 견적 발송 | ~14일 |
| 계약 | `WON` | 구매 확정 | 결제 or 계약서 서명 | 종료 |
| 이탈 | `LOST` | 거절 or 장기 무반응 | 3회 시퀀스 무반응 or 명시적 거절 | 종료 |
| 장기 육성 | `NURTURE` | 지금은 아니지만 나중에 가능 | "지금은 아니다" 회신 or 관리자 판단 | 3개월 |

### 스테이지 전환 규칙 (자동 vs 수동)

| 전환 | 자동/수동 | 트리거 |
|------|----------|--------|
| POOL → PROFILED | 자동 | 크롤 + 분석 완료 |
| PROFILED → LEAD | 자동 | 종합 60점+ & (시그널 감지 or 관리자 지정) |
| LEAD → CONTACTED | 자동 | 이메일 발송 성공 (Resend webhook) |
| CONTACTED → RESPONDED | 자동 | 오픈/클릭/회신 이벤트 |
| RESPONDED → DEMO | 반자동 | 데모 신청 폼 or 관리자가 수동 전환 |
| DEMO → PROPOSAL | 수동 | 관리자가 견적서 발송 |
| PROPOSAL → WON | 수동 | 관리자가 계약 확정 |
| ANY → LOST | 자동/수동 | 3회 시퀀스 무반응(자동) or 거절 회신(수동 확인) |
| LOST → NURTURE | 수동 | 관리자가 "장기 육성"으로 전환 |
| NURTURE → LEAD | 자동 | 3개월 후 재활성화 트리거 |

## 2. 스코어링 — 이 병원이 돈이 되는가

### 2-1. 병원 4축 분석 (Hospital Score)

> 목적: "이 병원이 장비를 살 만한 곳인가?"

| 축 | 가중치 | 핵심 질문 | 데이터 소스 |
|---|--------|----------|-----------|
| 투자성향 (Investment) | 35% | 장비에 돈을 쓰는 병원인가? | 보유 장비 종류, 수량, 고가장비 비율, 최근 도입 여부 |
| 포트폴리오 (Portfolio) | 25% | 시술이 다양하고 깊은가? | 시술 종류, 카테고리 분포, 리프팅 비중 |
| 시술규모 (Scale) | 25% | 환자가 많고 매출이 큰가? | 시술 수, 가격 공개 여부, 평균 단가, 의사 수 |
| 마케팅 (Marketing) | 15% | 마케팅에 투자하는가? | 웹사이트 페이지 수, 블로그/SNS 활동, 광고 집행 |

**등급 산출:**

| 등급 | 점수 범위 | 의미 | 이메일 전략 |
|------|----------|------|-----------|
| PRIME | 85~100 | 최우선 타깃, 즉시 영업 | 최고급 맞춤 이메일 (Claude Sonnet) |
| A | 70~84 | 핵심 타깃, 적극 영업 | 맞춤 이메일 (Claude Sonnet) |
| B | 55~69 | 일반 타깃, 표준 영업 | 세미 맞춤 이메일 (Claude Haiku) |
| C | 40~54 | 보류, 조건부 영업 | 범용 이메일 템플릿 |
| D | 0~39 | 제외, 영업 대상 아님 | 발송 안 함 |

**점수 산출 근거가 반드시 표시되어야 하는 이유:**
- "투자성향 70점"만 보면 → "그래서 뭐?"
- "투자성향 70점: 울쎄라+인모드+피코레이저 보유, 장비 총 19종, 최근 2년 내 3종 신규 도입, 적극 투자형" → "아, 이 병원은 장비에 돈을 아끼지 않는구나. TORR RF 설득 가능하겠다"
- **점수 근거가 곧 영업 화법의 재료**

### 2-2. 제품 매칭 분석 (Sales Angle Score)

> 목적: "이 병원에 TORR RF를 어떤 각도로 제안해야 하는가?"

TORR RF는 하나의 장비지만, 병원 상황에 따라 **제안하는 포인트가 달라야** 한다.

| 앵글 | 코드 | 가중치 | 타깃 병원 유형 | 핵심 메시지 |
|------|------|--------|-------------|-----------|
| Bridge 치료 | `bridge` | 45% | 울쎄라/HIFU 보유 | "울쎄라 사이 유지 시술로 환자 재방문율 ↑" |
| 수술후 관리 | `postop` | 25% | 성형외과 겸업 | "수술 후 빠른 회복 + 추가 매출" |
| 남성 리프팅 | `mens` | 15% | 남성 시술 비중 높음 | "남성 환자 잡는 비침습 리프팅" |
| 무통 리프팅 | `painless` | 10% | 통증 민감 환자 다수 | "통증 없는 리프팅으로 신규 환자 유입" |
| 바디 타이트닝 | `body` | 5% | 체형 시술 비중 높음 | "바디 타이트닝 메뉴 추가" |

**매칭 적합도 계산 예시 (815의원):**

```
815의원 현황:
- 울쎄라, 인모드, 슈링크 보유 → Bridge 적합도 높음
- 성형외과 아님 → 수술후관리 적합도 낮음
- 리프팅 시술 5종 → 리프팅 수요 확인
- 남성 시술 언급 없음 → 남성 리프팅 적합도 중간
- 바디 시술 없음 → 바디 적합도 낮음

결과:
- Bridge: 85% ← 최적 앵글
- 무통 리프팅: 68%
- 수술후 관리: 35%
- 남성 리프팅: 42%
- 바디: 12%

→ 이메일 피치: "울쎄라 시술 사이 유지 관리용 RF"로 접근
```

### 2-3. 시그널 감지

> 목적: "이 병원에 지금 연락해야 하는 이유가 있는가?"

스코어만으로는 "언제" 연락할지 모른다. 시그널이 "지금"을 알려준다.

| 시그널 | 감지 방법 | 영업 의미 | 긴급도 |
|--------|----------|----------|--------|
| 새 장비 추가 | 크롤 비교 (이전 vs 현재) | "장비 투자 중 → 추가 장비 고려 가능" | 🔴 높음 |
| 장비 노후화 | 도입 연도 3년+ 추정 | "기존 장비 교체 시기" | 🟡 중간 |
| 시술 메뉴 변경 | 크롤 비교 | "메뉴 개편 중 → 새 장비 니즈" | 🔴 높음 |
| 가격 인상 | 가격 비교 | "매출 성장 or 고급화 전략" | 🟡 중간 |
| 웹사이트 리뉴얼 | 구조 변경 감지 | "마케팅 투자 중 → 사업 확장" | 🟡 중간 |
| 의사 추가 | 의료진 페이지 변경 | "병원 규모 확장" | 🔴 높음 |
| RF 장비 없음 | 장비 목록에 RF 없음 | "RF 공백 → 핵심 타깃" | 상시 |

**시그널은 리드 자동 생성의 트리거:**
- 기존 PROFILED 병원에서 시그널 감지 → 자동으로 LEAD 스테이지로 전환
- 관리자에게 알림: "815의원에서 새 장비 추가 감지. 리드로 전환할까요?"

## 3. 이메일 시퀀스 — AI가 영업사원처럼 접근하는 법

### 3-1. 시퀀스 구조

하나의 리드에 대해 최대 3회 이메일을 보낸다. 각 이메일은 이전 반응에 따라 내용이 달라진다.

```
시퀀스 1 (D+0): 첫 접촉
  → 병원 분석 기반 맞춤 제안
  → "원장님 병원의 시술 라인업을 보고 연락드립니다"

  3일 후 반응 없으면 ↓

시퀀스 2 (D+3): 가치 제안
  → 경쟁 환경 + ROI 데이터
  → "강남구 피부과 5곳 중 RF 장비 보유는 1곳뿐입니다"

  5일 후 반응 없으면 ↓

시퀀스 3 (D+8): 마지막 터치
  → 긴급성 + 사회적 증거
  → "이번 달 도입 시 특별 조건 / 유사 병원 도입 사례"

  7일 후 반응 없으면 ↓

LOST (또는 NURTURE로 수동 전환)
```

### 3-2. 반응에 따른 분기

| 반응 | 의미 | AI 대응 |
|------|------|--------|
| 오픈만 | 관심은 있지만 확신 없음 | 다음 시퀀스에 임상 데이터/성공 사례 추가 |
| 오픈 + 클릭 | 높은 관심 | 즉시 팔로업: "더 궁금하신 점 있으시면..." |
| 오픈 + 데모 페이지 방문 | 구매 의향 | 자동 데모 안내 메일 |
| 회신 (긍정) | 확정적 관심 | 🔥 핫리드 알림 → 카카오톡 전환 제안 |
| 회신 (질문) | 정보 더 필요 | AI 1차 답변 초안 생성 → 관리자 검수 → 발송 |
| 회신 ("지금은 아니다") | 타이밍 아님 | 정중 마무리 + NURTURE(3개월 후 재접근) |
| 회신 ("필요 없다") | 관심 없음 | 마무리 + LOST |
| 수신거부 | 법적 거부 | 즉시 리스트 제거 (법적 필수) |
| 3회 무반응 | 무관심 | LOST → 6개월 후 재평가 |

## 4. Admin 화면 설계 — 각 화면의 목적과 행동

### 4-0. 설계 원칙

모든 화면은 이 3가지를 충족해야 한다:

1. **목적**: 이 화면에서 뭘 보는가?
2. **판단**: 이 화면에서 어떤 결정을 내리는가?
3. **행동**: 판단 후 뭘 하는가? (버튼, 링크, 필터 등)

---

### 4-1. 📊 대시보드 — "지금 시스템이 어떤 상태인가?"

**목적**: 로그인 후 3초 안에 전체 상황 파악
**판단**: 어디에 병목이 있는가? 오늘 내가 할 일은?
**행동**: 병목 지점 클릭 → 해당 페이지로 이동

## 6. 전화번호 포맷팅 규칙

심평원 원본 데이터는 하이픈 없이 저장됨. 표시할 때만 포맷.

```typescript
function formatPhoneNumber(raw: string): string {
  const cleaned = raw.replace(/\D/g, '');

  // 서울 (02)
  if (cleaned.startsWith('02')) {
    if (cleaned.length === 9) return cleaned.replace(/(\d{2})(\d{3})(\d{4})/, '$1-$2-$3');
    if (cleaned.length === 10) return cleaned.replace(/(\d{2})(\d{4})(\d{4})/, '$1-$2-$3');
  }

  // 휴대폰 (010, 011, 016, 017, 018, 019)
  if (cleaned.startsWith('01')) {
    if (cleaned.length === 10) return cleaned.replace(/(\d{3})(\d{3})(\d{4})/, '$1-$2-$3');
    if (cleaned.length === 11) return cleaned.replace(/(\d{3})(\d{4})(\d{4})/, '$1-$2-$3');
  }

  // 지역번호 (031~064)
  if (cleaned.length === 10) return cleaned.replace(/(\d{3})(\d{3})(\d{4})/, '$1-$2-$3');
  if (cleaned.length === 11) return cleaned.replace(/(\d{3})(\d{4})(\d{4})/, '$1-$2-$3');

  return raw; // 포맷 불가 시 원본 반환
}
```
